# 文档规范 · Documentation Standards

> 目标：让文档成为可信、可追溯、活的事实来源，支撑规范驱动开发。

## 1. 文档分层与职责

| 文档 | 面向 | 职责 | 变更频率 |
|------|------|------|----------|
| `.specify/memory/constitution.md` | 全体 | 不可变最高原则 | 极低（需修订记录） |
| `AGENTS.md` | AI | 协作规则/命令/边界（事实来源） | 随工程约定 |
| `CLAUDE.md` | Claude | 薄封装，指向 AGENTS.md | 低 |
| `.cursor/rules/*.mdc` | Cursor | 作用域规则 | 随约定 |
| `docs/prd/` | 全体 | 产品需求（做什么/为什么） | 需求变更时 |
| `docs/tech-spec/` | 研发 | 技术设计（怎么做） | 设计变更时 |
| `docs/standards/` | 全体 | 工程规范（含 Skill 设计方法论 `skill-design.md`） | 低 |
| `docs/process/` | 全体 | 流程说明 | 低 |
| `skills/` | AI/研发 | 项目自定义 Skill（能力包） | 按需 |
| `specs/NNN-*/` | 全体 | 单功能规格/计划/任务 | 功能开发期 |
| `README.md` | 人类 | 项目介绍与上手 | 中 |

## 2. 核心原则

- **单一事实来源**：同一信息只在一处权威定义，其他地方引用而非复制。
- **活文档**：行为/约定变化时，先更新文档再改代码（宪法第二条）。
- **精简**：root 级 AI 文档 < 300 行；spec 保持 1–3 页，过大则拆分。
- **可追溯**：代码/提交/PR 关联对应 spec；决策记录留档。
- **点引用而非内嵌**：大文档用路径引用（如「见 `docs/tech-spec/TECH_SPEC.md`」），避免重复膨胀。

## 3. 写作约定

- 中文为主，术语可保留英文（如 EARS、SM-2、Conventional Commits）。
- 使用清晰的标题层级（≤ 3 级）、表格与列表，便于人与 AI 解析。
- 每份文档在头部标注：版本、状态、最后更新日期、上游关联文档。
- 验收标准统一用 **EARS 语法**（见 `docs/process/spec-driven-development.md`）。

## 4. 维护

- 文档变更纳入同一 PR，与相关代码一起评审。
- 宪法变更须在其「修订记录」表追加条目。
- 定期回顾：发现文档与实现不一致时，以「先改文档」的方式纠正。
