# 测试规范 · Testing Standards

> 精简作用域版见 `.cursor/rules/testing.mdc`。测试框架：Vitest（前后端统一）。

## 1. 必测范围（宪法红线）

以下核心逻辑**必须**有单元测试，且随实现一起提交：

- **SM-2 间隔重复算法**：各评分（Again/Hard/Good/Easy）分支下 `ease_factor`、`interval_days`、`repetitions`、`due_date` 的计算，含边界（EF 下限 1.3、首/次复习）。
- **打卡/坚持统计**：按天聚合、累计天数、今日进度、跨天与时区边界。
- 任何纯函数 / 工具函数 / 数据转换逻辑。

## 2. 测试策略（分层）

- **单元测试**：纯逻辑与工具函数，占比最高，快、无外部依赖。
- **集成测试**：API 路由 + 数据访问，用内存 SQLite，验证端到端数据流。
- **组件测试**（可选）：关键交互组件（翻卡、评分、编辑器粘贴图片）。
- MVP 阶段以单元测试为主，优先覆盖高价值/高风险逻辑，不盲目追求覆盖率数字。

## 3. 编写原则

- 核心逻辑走 **TDD**：先写失败测试，再实现，再重构。
- 结构遵循 **AAA**：Arrange（准备）→ Act（执行）→ Assert（断言）。
- 一个测试只验证一个行为；测试名清晰描述「场景 → 期望」。
- 测试独立、可重复、无顺序依赖；不依赖真实网络/磁盘/时钟（必要时注入时间）。

## 4. 命名与位置

- 测试文件与被测文件同名相邻：`srs-algorithm.ts` → `srs-algorithm.test.ts`。
- 或集中于就近的 `tests/` 目录，团队内保持一致。

## 5. 边界覆盖清单（记忆类应用重点）

- 空数据 / 无到期卡片的空状态。
- 新卡首次、第二次复习的间隔跳变。
- 连续正确与中途 Again 的间隔重置。
- 跨天（午夜边界）、时区归日一致性。
- 连续打卡与中断后的累计统计。

## 6. 运行

```bash
npm test            # 全量
npm test -- --watch # 监听模式
```

提交涉及核心逻辑的变更前，`npm test` 必须通过。
